## 桌面端故障复现

- 页面：`/zh/blog/llm-post-training-basics-and-jargon/`
- 视口：1280 × 900px
- 修复前连续发送 5 次 `500px` 向下滚轮事件，页面位置为 `0 → 342 → 321 → 525 → 327 → 321`，无法持续向下阅读。
- 原因：根元素上的 `overflow-x: hidden` 形成额外滚动容器；目录高亮使用 `scrollIntoView()` 时会连带调整文档滚动位置。

## 移动端二次复测

- 第一版回归只校验了根元素宽度与按钮坐标；根级 `overflow-x: clip` 将 `documentElement.scrollWidth` 压到 390px，但 `body.scrollWidth` 仍为 701px，因此不能代表所有真实手机浏览器都不会采用被撑宽的页面坐标系。
- 进一步定位到 KaTeX 的屏幕阅读器 MathML 层仍会把文章内容行撑宽。移动端现由 `.post-content-row` 在内容层限制横向溢出，不再依赖根元素掩盖问题；公式、表格和代码块仍在各自容器内横向滚动。
- 移动端测试新增强制条件：`body.scrollWidth` 必须等于设备视口宽度。

- 两篇目标文章在 320、360、375、390、412、430、667、844、932、1023px 下共验证 20 组，全部通过。
- 每组移动端测试均校验：根布局视口等于设备视口、目录按钮距屏幕右侧 12px、打开后的目录面板右边缘等于屏幕右边缘。

## 手机目录球动态视口矫正

- 日期：2026-07-22
- 根因一：旧定义在正文底部刚进入视口底边时就记为 100%，此时最后一屏正文仍未读完；第一轮矫正沿用了这个错误终点，旧测试也把它当作正确答案。
- 根因二：旧公式只使用 `window.innerHeight`（布局视口），真实手机的地址栏、软键盘或缩放会单独改变视觉视口，因此圆环可能提前或滞后。
- 根因三：旧测试把代码自己写入的 `stroke-dashoffset` 当成期望值，没有根据 SVG 的真实圆周长度校验肉眼看到的弧长，也没有覆盖 `visualViewport` 与 `innerHeight` 不等的场景。
- 根因四：手机目录按钮自身的完整边框、半透明背景边缘和阴影都紧贴进度路径。32px 按钮与原 36px SVG 中 `r=22` 的圆形路径中心线半径只相差约 0.5px；即使移除边框，按钮轮廓和阴影仍会出现在 SVG 未绘制的弧段下方，导致真实进度只有约 75% 时肉眼仍像满环。桌面进度条不受此问题影响。
- 修正：100% 改为正文底部到达页头下方阅读线；若页面无法滚到该理想终点，则以视觉视口能到达的页面最底部为终点。阅读坐标使用 `visualViewport.pageTop/height`，监听视觉视口的 `scroll` / `resize`，并在正文尺寸变化时通过 `ResizeObserver` 重算；圆环用 `getTotalLength()` 的实测周长设置虚线和偏移。为避免亚像素缺口被抗锯齿显示成闭环，真实进度未到 100% 时把可见弧长上限设为 98%；按钮移除边框、阴影和半透明背景，进度环按最终视觉要求保持 36px、1px 描边并贴近 32px 球体。
- 纯函数单测 `7/7` 通过。项目原始 `npm run dev` 启动链路下的 Chrome 浏览器回归覆盖桌面 `1280×900`、手机 `390×844`、布局视口 `844px` / 视觉视口 `600px` 三种场景，共 `3/3` 通过；正文底部仅刚进入视口时，普通手机真实进度为 `0.9926`，动态视觉视口场景为 `0.9940`，两者的可见圆环均保持为 `0.98`，36px 环上的缺口约 `2.06px`；在 `Reward model 与偏好概率` 标题处，390px 手机真实进度、DOM 与 SVG 描边均为约 `0.7504`，定点回归确认圆弧保持开环；视觉视口高度与顶部偏移变化后的 DOM 和圆环进度均与独立期望值一致。
- Astro 构建成功，生成 45 个页面；外部友链 feed 在受限网络中出现非致命 fetch warning。

验证脚本：`tests/reading-progress.test.mjs`、`tests/inspect-reading-progress.mjs`
