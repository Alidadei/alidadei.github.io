# 网站防复制与安全边界

更新日期：2026-07-19

## 结论

公开发布的 Astro 静态站点无法从技术上做到“别人拿到前端后完全不能复制或改造”。浏览器必须取得 HTML、CSS、JavaScript 和公开资源；任何写在前端的域名判断、密钥校验或停用逻辑，复制者都能删除。

能真正建立边界的是：不公开源代码、敏感能力只放服务端、服务端独立鉴权、密钥不进入仓库。即使源码仓库改为私有，已经发布到公网的页面与文章仍可被抓取，因此内容保护还需要明确权利声明、来源标记和侵权处置。

## 已落地的服务端隔离

- `worker/wrangler.toml` 只允许 `https://alidadei.github.io`、带端口的 `localhost` 和 `127.0.0.1` 开发来源。
- Worker 在路由处理前拒绝其他浏览器 `Origin`，返回 403，不再反射任意来源。
- CORS 只限制浏览器跨域读取，不等同于鉴权；所有敏感 API 仍必须保留现有会话校验和 GitHub 用户白名单。
- `tests/worker-cors.test.mjs` 锁定官网、本机、仿冒子域名、陌生克隆站及无 `Origin` 请求的行为。
- 代码修改尚未部署；只有重新部署 Cloudflare Worker 后，线上行为才会改变。

## 按效果排序的后续措施

1. 如果目标是“不让别人直接取得源码并运行”，把源仓库改为私有，由私有 CI 只发布构建产物。这是最有效的源码控制；公开页面本身仍可被抓取。
2. 把 GitHub App secret、允许用户 ID 和所有令牌继续放在 Cloudflare/GitHub Secrets 中，不写入前端、`wrangler.toml` 或提交历史；发现泄露应轮换，而不是只从最新提交删除。
3. 在确认作者名称、适用文件和第三方代码授权后，可增加单独的 `COPYRIGHT.md`。不要用一个笼统声明覆盖第三方材料；现有 README 已注明 3D 背景来源。
4. 保留 canonical、作者页脚、提交历史和原始发布时间。它们不能阻止复制，但能形成来源证据并方便搜索引擎识别原始页面。
5. 定期用文章中特征句、站点名和图片反向搜索监测复制；确认侵权后先保存 URL、截图和时间证据，再联系托管方或按平台流程提交移除请求。

## 不应当采用

- 禁止右键、禁止选择或屏蔽快捷键：不影响下载、查看源代码或自动抓取，只损害正常访问。
- JavaScript 域名锁、前端构建密码、远程“自毁开关”：逻辑与校验都随源码公开，可以被删除，也可能误伤本站。
- 代码混淆作为安全措施：只能增加少量阅读成本，不能保护公开内容或服务端凭据。
- 隐藏水印冒充访问控制：水印适合举证，不构成技术权限边界。
