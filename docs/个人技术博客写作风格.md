# 个人技术博客写作风格

本文用于沉淀个人技术博客的写作习惯，主要依据 `LLM 后训练基础概念阐述.md` 的人工修改前后 diff，并参考《Temperature 的数学本质》《LLM RL 中数据复用引发的 off-policy 问题与解决方法》等既有文章总结。

## 文章结构

1. **先讲写作动机**：从自己学习或实践中真正遇到的困惑切入，说明为什么值得写，而不是上来直接罗列定义。
2. **先给全局，再拆概念**：先用流程图或几句话交代全貌，让读者知道每个概念处在什么位置，再逐个展开。
3. **围绕主问题取舍内容**：只保留理解当前主题真正需要的知识；与主线关系较弱的百科式扩展主动删掉，复杂推导可以引到已有文章。
4. **按概念依赖关系排序**：先解释输入、概率、采样等前置概念，再解释 policy、reward、advantage 和训练算法，避免用尚未定义的黑话解释另一个黑话。
5. **结尾做关系归纳**：优先使用对照表、阅读配置时的问题清单或三句话总结，帮助读者重新把概念连起来。

## 语言风格

- 使用第一人称交代观察和判断，例如“我在实践中发现”“个人更倾向于”。
- 整体使用大白话，允许“黑话、吐出、对得上号、钻空子、限速器”等口语或比喻，让抽象概念先变得可感知。
- 句子习惯按“现象 → 原因 → 结果”展开，常用“其本质是、因此、不过、因而、换句话说”串起逻辑。
- 第一次出现技术名词时尽量同时给出中文、英文和常用符号，后文再使用简称。
- 主观看法与事实分开：论文或官方文档支持的内容直接陈述；存在口径差异时明确写“没有统一口径”；个人取舍用“个人认为/本文采用”标识。
- 不使用空泛的讨好话，也不为了显得严谨而堆叠无助于理解的保留意见。

## 概念解释模板

一个重要概念优先按下面的顺序来写：

1. **一句话定义**：它到底是什么。
2. **必要的公式**：把比较对象、优化对象或约束对象写清楚。
3. **逐个解释符号**：不能让公式成为装饰。
4. **翻译成大白话**：用抽奖、山峰、模拟试卷等读者熟悉的事物建立直觉。
5. **给出实际效果**：参数调大或调小会怎样，训练日志里会看到什么。
6. **说明边界和易错点**：它不是什么、不能推出什么、不同框架哪里可能口径不同。
7. **需要时链接旧文**：当前文章说清主线，复杂推导交给专题文章。

可压缩成一句话：

```text
定义 → 公式 → 符号 → 大白话直觉 → 工程效果 → 边界
```

## 常用呈现方式

- 用 `text` 代码块画训练链路、数据流或因果关系。
- 抽象角色第一次出现时，立即落到真实训练对象：它对应哪一份模型、checkpoint、模型副本或日志张量，是否更新，以及在什么时刻固定下来。
- 示意图必须直接画出正文正在比较的全部对象及其关系。正文比较 $P$、forward KL 下的 $Q$ 和 reverse KL 下的 $Q$，图里就不能只画 $P$。
- 用一个具体数字例子解释概率、ratio、batch 等量，而不是只给抽象定义。
- 用粗体突出每段真正需要记住的一句话，不把整段都加粗。
- 多组容易混淆的概念用表格做“最短区分”。
- 公式之后必须紧跟语言解释；如果读者跳过公式，也应能继续读懂正文。

## 修改时需要防止的问题

- 口语化不能牺牲准确性。例如 logit 可以粗略叫“匹配打分”，但不能直接等同于余弦相似度。
- 不把个人采用的术语边界写成全行业标准。
- 不使用没有来源的绝对结论，例如“某参数一定提升准确率”。
- 示例控制 token、变量名和框架参数如果只是示意，必须明确标注，避免读者误以为是真实协议。
- 长句可以保留个人节奏，但一个句子同时塞入过多转折时应拆开，否则因果关系会变得模糊。
- 删除正文后同步清理失去引用关系的参考资料，避免参考文献看起来很多、正文却没有使用。
- 所有批注、空公式、未解释变量和临时占位词必须在成稿前清零。

## 成稿检查清单

- [ ] 开头是否说明了“为什么写”。
- [ ] 读者能否先看到全局关系，再进入细节。
- [ ] 每个黑话是否都有一句大白话解释。
- [ ] 每个公式的变量、方向和统计单位是否明确。
- [ ] 事实、行业口径和个人观点是否分开。
- [ ] 参数效果是否同时写了目的、现象和边界。
- [ ] 是否删掉偏离主线或与旧文重复的内容。
- [ ] 引用是否真的支持附近的表述。
- [ ] 文中是否还残留引号批注、空公式或占位词。

## 二次修改反思（2026-07-17）

本轮在人工继续修改《LLM 后训练基础概念阐述》后，再次对照 diff，可以补充几条更细的偏好：

- **相关对比尽量就近出现**：例如 entropy 和 temperature 的区别，与其留在 Entropy 章节末尾，不如放到 Temperature 解释之后。读者刚看到第二个概念时立刻做对比，记忆成本更低。
- **公式引入要直接**：比起先说一大段再放公式，更偏好“由这些量做线性计算得：”然后紧跟公式，让语法直接落到数学表达上。
- **讲清即止，不追求把所有后果列完**：特殊 token 的作用已经说明后，可以删去过长的 BOS/EOS 重复后果枚举。正文应回答当前概念问题，而不是沿每个分支无限扩写。
- **相邻短段可以合并**：当两段共同回答同一个因果问题时，合并成一段更符合个人连续叙述的节奏，也能减少文章被切得过碎。
- **章节开头需要一句定位**：倾向用“这一部分……”先告诉读者接下来讲哪个转换过程，再进入公式或步骤。

同时，反思时要区分“稳定偏好”和“编辑中的临时痕迹”。单次出现的标题层级丢失、重复标点等现象不能沉淀成风格规则，应留给成稿检查处理。
