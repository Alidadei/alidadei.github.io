## 反思修改

### 核心编辑边界

- 用户已经手动写好或修改过的内容是当前基线，必须保留，不能借“完善、润色、纠错”顺手重写。
- 开始修改前先查看 `git diff`，识别并保护用户亲自改过的段落。
- 只修改用户明确点名、批注或圈定的范围；没有被要求修改的正文、结构、标题、措辞和例子一律不动。
- 不得擅自恢复用户已经删除的内容，也不得因为个人判断而重新排序、扩写或压缩其他章节。
- 如果发现未授权范围存在事实、语法或排版问题，只向用户报告，不直接修改。
- 如果完成指定修改不可避免地需要调整相邻内容，只做维持上下文成立所需的最小改动，并明确说明。
- 修改完成后再次查看 `git diff`，确认改动没有越出用户授权范围。

### 反思与提交

1. 通过 `git diff` 对比本人修改前后的文章差异，总结本人的写作思路、语言风格、句子结构、语法顺序、内容取舍和用词偏好。
2. 区分稳定的写作偏好与编辑过程中的临时痕迹，不能把错别字、重复标点、标题层级遗漏等问题总结成个人风格。
3. 将可复用的结论沉淀到 `docs/个人技术博客写作风格.md`；若项目已有其他对应的风格或经验文档，则优先合并，不重复创建。
4. 完成反思和经验沉淀后，**必须立即创建一次 Git commit，作为下一轮人工修改的对比基线**：
   - 先用 `git status --short` 确认工作区范围；
   - 只暂存本轮反思涉及的文件，不夹带无关改动；
   - 提交前检查 `git diff --cached --stat` 和 `git diff --cached --check`；
   - commit 完成后再次检查 `git status --short`，并向用户报告 commit hash。
5. 除非用户明确要求，否则只 commit，不 push。
