# 知识架构分类(设计说明)

> 本文是知识三轴的**设计 rationale**——为什么这么分。字段格式、可用 slug 值、构建校验、`new-post` 脚手架等**操作规则**见 `技术博客排版规范.md`;视图实现见 `knowledge-graph-en/knowledge-graph.md`。

## 三轴:origin × subject × maturity

每篇文章用三个**正交**维度标注,各管一摊、互不重叠:

- **origin**(产出方式):我怎么得到这个知识——学习还是动手。→ 存 `categories[0]`。
- **subject**(知识主题):关于什么——transformer、agent、数据结构……。→ 存 `knowledge`。
- **maturity**(时效定位):在时间轴上的位置——基础 / 当下热点 / 未来展望。→ 存 `maturity`。

三者垂直:origin 是「怎么来的」,subject 是「关于什么」,maturity 是「现在算基础还是热点」。它们不是一棵树里要划分的几段,而是同一篇文章的三个独立标签——可以自由交叉(如「Agent」这个 subject 下,既有 origin=学习的理论文,也有 origin=实践的动手文)。

## 为什么 maturity 独立成轴(防腐烂)

把 maturity 当 subject 的子节点(如「基础 / 大模型 / 未来」作为主题节点)会让主题树**腐烂**:2025 的「当下热点」≠ 2026 的「当下热点」,却共用一个节点名,越积越乱。

抽成独立维度、锚在文章 `date` 上就不腐烂——读者看日期就懂「2025-06 标当下热点」指什么,无需额外字段。subject(主题)本身永不过时(transformer 永远是 transformer)。

## 沉淀(graduation)

前沿(maturity=`当下热点`)里的概念,经时间检验成为奠基性 → 把相关文章的 maturity 从 `当下热点` 改成 `基础`(**subject 不动**)。这是 maturity 维度的重标注,不碰主题树。

定期(如每季度)扫一遍当下热点,问「是否已奠基、几年后仍成立」,是则沉淀。

## 重构信号

- 某 subject 节点 > ~8 篇且内部聚类 → 拆节点(如 Agent → Agent 理论 / Claude Code)。
- 全新方向 → `knowledge.json` 加节点。
- 未来展望里的方向成熟 → maturity 迁到 基础 / 当下热点。
