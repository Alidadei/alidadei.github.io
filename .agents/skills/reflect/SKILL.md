# 反思修改并提交

严格按顺序执行。将用户在显式调用 `$reflect` 时附带的文件路径和要求视为本轮输入；路径包含空格时，按真实的完整路径处理，不能凭空拆分或猜测。

## 1. 确定范围

1. 完整读取 `docs/DRAFT/反思修改.md`，将它作为本流程的当前规则源。
2. 读取 `docs/个人技术博客写作风格.md`（若存在），避免重复沉淀已有结论。
3. 运行 `git status --short`、`git diff --name-status HEAD` 和 `git diff --stat HEAD`，确认工作区全貌。
4. 优先使用用户输入指定的目标文档。若没有指定，仅在工作区中只有一个明确的人工编辑目标时才自行确定；存在多个候选目标时，停止并请用户指定，不能猜。
5. 若目标文件未被 Git 跟踪，说明缺少可比较的历史基线；得到用户确认前，不把整篇新文件误当成“修改偏好”的证据。

## 2. 保护用户内容

- 将用户已经写好或修改后的内容视为当前基线。反思过程默认不改目标文档。
- 只有用户在本次调用中明确点名、批注或圈定并要求修改的范围才允许改；其他正文、结构、标题、措辞和例子一律不动。
- 不恢复用户删除的内容，不顺手润色、纠错、重排、扩写或压缩未授权部分。
- 在未授权范围发现事实、语法或排版问题时，只报告，不修改。
- 指定修改必须牵连相邻内容时，只做保证上下文成立所需的最小改动，并明确报告。

## 3. 根据 diff 反思

1. 对每个目标文件运行 `git diff HEAD -- <准确路径>`，同时覆盖已暂存和未暂存的变化。
2. 从实际的“修改前 → 修改后”证据中总结：
   - 写作思路与内容取舍；
   - 概念和章节的组织顺序；
   - 句子结构、语法顺序和段落节奏；
   - 用词、语气、解释深度和例子偏好。
3. 每条结论都要能指向具体 diff。没有证据就不下结论；证据较弱时标记为暂定偏好。
4. 区分稳定偏好与临时编辑痕迹。错别字、重复标点、残句、标题层级遗漏等不能总结成个人风格。
5. 先向用户给出简明反思结论，再进行经验沉淀和提交。

## 4. 沉淀经验

- 仅将新增、稳定、可复用且有 diff 支持的结论最小化合并到 `docs/个人技术博客写作风格.md`。
- 保留原文结构与用户已有表述；能补一条就不重写一节，已有同义结论则不重复添加。
- 如果没有新的稳定结论，不为制造改动而修改风格文档。
- 修改后用 `git diff -- <风格文档路径>` 检查沉淀范围，确认没有越界。

## 5. 安全提交

1. 再次运行 `git status --short`，列出本次应提交的精确文件：目标文档、确有更新的风格文档，以及用户明确纳入本轮的其他文件。
2. 提交前检查暂存区。若其中已有不属于本轮的文件，停止并报告；不得把无关改动带入 commit，也不得擅自清空用户的暂存区。
3. 只用精确路径暂存本轮文件，不使用 `git add .`、`git add -A` 或宽泛通配符。
4. 运行 `git diff --cached --name-only`、`git diff --cached --stat` 和 `git diff --cached --check`。暂存文件集合与预期不完全一致时，不提交。
5. 根据实际改动拟定简洁 commit message 并创建 commit。若目标没有 diff，不创建空提交。
6. commit 后运行 `git status --short` 和 `git rev-parse --short HEAD`，向用户报告 commit hash、已提交文件和仍未提交的改动。
7. 只 commit，不 push，除非用户明确要求 push。

## 6. 最终自检

- 确认没有修改用户未授权的正文。
- 确认反思结论都有 diff 证据。
- 确认 commit 不含无关文件。
- 如果任一项无法确认，停止提交并把具体阻碍告诉用户。
